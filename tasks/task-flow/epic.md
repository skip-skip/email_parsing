## Architecture

### Missing Info Flow
```
Current (broken):
  Approve → sets status="APPROVED" → no email sent → item disappears

Target:
  Approve → send_reply_all() → status="AWAITING_REPLY" → shows in second list
                                                          ↓
  Reply arrives → detect conversation_id match → ConversationHandler.handle_reply()
    → merge fields → re-validate → persist to DB
      → if complete → transition to READY_FOR_SCHEDULING → remove from queue
      → if still missing → re-queue with updated draft
```

### Scheduling Queue Flow
```
Current (broken):
  Approve → sets APPROVED=True → no email sent, no calendar created
  Decline → sets DECLINED=True → no email sent

Target:
  Approve → send_reply_all(acceptance) → create_event() per block → transition to CALENDAR_CREATED
  Decline → send_reply_all(decline + user reason) → delete_event() if exists → transition to WAITING_FOR_INFORMATION
```

### Task Closure Flow
```
Current: No close functionality exists

Target:
  Close Task button → confirmation modal → transition to COMPLETED → draft completion email → send via send_reply_all()
  Closed Tasks page → list completed/archived tickets
```

### Workflow Pipeline Flow
```
Current (stubs):
  receive_email → extract_task → validate_fields → plan_schedule (stub) → create_calendar_event (stub) → dispatch_task (stub) → END

Target:
  receive_email → extract_task → validate_fields
    → plan_schedule (real: CalendarPlanningAgent → SchedulingQueue) → END (pending user approval)
    → draft_missing_info_email (real: EmailDraftAgent → MissingInfoQueue) → END (pending user approval)
    → create_calendar_event (real: OutlookComCalendarProvider → CalendarEvent DB)
    → dispatch_task (real: transition to READY_FOR_SCHEDULING or WAITING_FOR_INFORMATION)
```

### State Machine Enforcement
```
Current (bypassed):
  EmailProcessor._persist_ticket_status() → ticket_repo.update_status() (no validation)

Target:
  EmailProcessor._persist_ticket_status() → transition_ticket() (validates against VALID_TRANSITIONS)
    → strict_mode=False: log warning but allow (backward compat)
    → strict_mode=True: raise error (for new code)
```

### Dashboard Flow
```
Current (static):
  DashboardPage → hardcoded zeros → no API calls

Target:
  DashboardPage → useQuery(api.tickets.listActive, api.missingInfo.list, api.scheduling.list)
    → real counts in 4 cards → clickable navigation
    → LLM health indicator → recent activity feed
```

## Acceptance Criteria
- Clicking "Approve & Send" actually sends a reply-all email via Outlook
- Approved items move to "Awaiting Reply" list (status=AWAITING_REPLY)
- Missing info drafts are generated by the LLM (EmailDraftAgent)
- When a reply arrives for a WAITING_FOR_INFORMATION ticket, it merges into the existing ticket
- If the reply provides all missing fields, the ticket moves to READY_FOR_SCHEDULING
- If fields are still missing, the ticket is re-queued with an updated draft
- The missing info queue page shows two tabs: "Pending Review" and "Awaiting Reply"
- Approving a schedule sends a confirmation email and creates Outlook calendar events
- Declining a schedule sends a decline email with user-entered reason
- Approving a schedule transitions ticket to CALENDAR_CREATED (appears on Active Tasks)
- Declining a schedule transitions ticket to WAITING_FOR_INFORMATION
- "Close Task" button works from TaskDetail page
- Closing a task sends a completion notification email to the client
- Closed Tasks page shows completed and archived tickets
- Dashboard shows real counts for all summary cards
- Dashboard cards are clickable and navigate to correct pages
- Active Tasks page supports pagination
- Tickets can be edited via PATCH endpoint and UI
- Workflow nodes perform real work (not stubs)
- State machine is enforced in EmailProcessor
- ConversationHandler persists merged fields to database
- Sidebar shows notification badges for queue counts
- Queue items link to ticket detail page
- No duplicate type definitions
- All existing tests continue to pass
